# Pre-launch Refresh of claude-code-router Runtime Config

## Summary

Make `.clauded.yaml` edits affecting the `claude_code_router` proxy take effect on the next `clauded` invocation, without requiring `clauded --reprovision`. Today, the rendered CCR config (`~/.claude-code-router/config.json`) and the custom router JS (`/etc/clauded/ccr-router.js`) are written only at provision time by the Ansible role; runtime YAML edits are silently ignored until the user re-runs the full playbook.

The fix moves the **config-rendering** responsibility into the host CLI as a per-launch step, while leaving the **install** responsibility (npm, wrapper script, role files) to provisioning. Users edit YAML, run `clauded`, and the proxy picks up the change immediately.

## Background

The current architecture renders CCR's runtime artifacts in two places that can only be triggered by Ansible:

1. **`config.json`** â€” generated by `roles/claude_code_router/tasks/main.yml::Render CCR config.json`. Contains the Provider list (with the live host-Ollama probe results), the `Router` block, and the path to the custom router JS.
2. **`ccr-router.js`** â€” generated by `roles/claude_code_router/tasks/main.yml::Render custom router JS`. Encodes the user's `overrides:` mapping as a small JS function.

Both files depend on `.clauded.yaml` content. Reprovisioning to apply a one-line YAML edit is heavy â€” full role replay, npm install reverification, host environment re-injection. It also wastes time when the user is iterating on routing behavior (overriding haiku to a different Ollama model, swapping curated providers, etc.).

A separate concern that this epic also fixes: the host's Ollama model list drifts independently of clauded. Users who `ollama pull` a new model after provisioning today must `clauded --reprovision` for the model to appear in CCR's Provider list. Re-probing at launch eliminates that workflow trap.

## Problem Statement

Routing changes that are purely *configurational* (no install, no system-package change) require a full provisioning run today. This breaks the declarative-config promise: edits to `.clauded.yaml` should take effect predictably without users needing to know which CLI subcommand applies which kind of change.

## Requirements

### R1 â€” Per-launch config refresh

When `vm.claude_code_router.enabled` is true, every `clauded` invocation that launches the harness MUST regenerate the CCR runtime artifacts (`config.json` and `ccr-router.js`) from the current `.clauded.yaml` and push them into the VM **before** the harness command runs.

The refresh MUST run on:

- The default `clauded` (with or without `--harness <name>`).
- Detection-driven flows (`clauded` against a project where `.clauded.yaml` already exists).

The refresh MUST NOT run when CCR is disabled in the config.

The refresh MUST NOT run on `clauded --stop`, `--destroy`, `--reboot`, `--force-stop`, or other lifecycle commands that don't launch the harness.

The refresh MAY run on `clauded --reprovision`, but is harmless there (Ansible re-renders the same files).

### R2 â€” Single source of truth for renderers

The Python implementation that renders `config.json` and `ccr-router.js` MUST be the same logic invoked by the Ansible role at provision time. Drift between the two renderers â€” e.g., Ansible Jinja and Python rendering producing different config for the same input â€” MUST be impossible by construction.

Acceptable strategies:
- A Python rendering function called both directly by the host CLI and by an Ansible action plugin (Ansible can invoke arbitrary Python).
- The host CLI is the only renderer; Ansible no longer renders these files at provision time and instead invokes the same Python pre-launch step.
- Either approach is fine; the requirement is that there is exactly one canonical implementation.

### R3 â€” Live host-Ollama model probe

The pre-launch refresh MUST probe the configured Ollama host (`host.lima.internal:11434` by default) at refresh time and include any newly available models in the rendered Provider entry. Models removed from the host MUST disappear from the rendered config.

Probe failure (host unreachable, timeout) MUST behave identically to today's role: emit a warning, render an empty Ollama models list, continue. The harness still launches; CCR's Anthropic and curated-provider routing is unaffected.

### R4 â€” CCR restart with new config

The wrapper script (`clauded-ccr-with`) already watches for token/credential changes and restarts CCR. The refresh step MUST trigger the same restart path so the proxy picks up the new config without manual intervention. Equivalent acceptable approaches: bumping a fingerprint file the watcher checks, sending the running CCR a signal, or relying on the watcher's existing restart-on-change semantics if a config-mtime check is added.

The refresh MUST NOT race with an in-flight CCR session; if a session is mid-stream, the restart MAY wait for the session to drain or restart cleanly between requests. (CCR is not part of in-flight conversation state.)

### R5 â€” Performance budget

The full refresh (Python render â†’ push into VM â†’ trigger CCR restart â†’ wrapper health-probe) MUST complete in under 3 seconds on a warm VM with a reachable Ollama host. The harness launch already waits on the wrapper's health probe today; the refresh budget is incremental on top of that.

If the refresh exceeds 3 seconds repeatedly, that's a regression to investigate, not a steady-state cost users should pay.

### R6 â€” Error visibility

A YAML syntax error, a schema validation failure, or a render failure MUST surface to the user as a clear actionable error before the harness launches. The harness MUST NOT launch with a stale CCR config when the new config is invalid.

The error message MUST name the offending file and (where possible) the offending line/key.

### R7 â€” Idempotence and re-rendering noise

Re-running `clauded` without YAML changes MUST be observable as a no-op: the rendered files are byte-identical to what's already in the VM, and CCR is not unnecessarily restarted. (This avoids gratuitous proxy bounces between adjacent invocations.)

Implementations MAY hash file contents and skip the push when unchanged.

### R8 â€” Backward compatibility

A first-time provision (clean VM, no CCR installed) MUST still produce a working CCR setup. Whether that's accomplished by Ansible rendering at provision time, or by Ansible installing the package and then deferring rendering to the first `clauded` launch, is an implementation decision. The user-visible outcome MUST be identical.

A user who provisioned under the current (Ansible-renders) implementation and then upgrades clauded MUST get the new pre-launch refresh behavior on the next `clauded` invocation, without needing to reprovision.

### R9 â€” Reprovision still works

`clauded --reprovision` MUST continue to be a valid operation that produces a known-good VM state, including a working CCR config. The refresh feature does not deprecate `--reprovision`; it eliminates the need to use it for *configurational* changes.

## Out of Scope

- **Refreshing CCR while a request is in-flight.** This epic accepts the existing wrapper-watcher restart semantics; mid-stream behavior is whatever the watcher already does. Live request migration is a separate, much larger feature.
- **Refresh of non-CCR roles.** Other roles (python, node, claude_code, opencode, etc.) continue to require `--reprovision` for changes. Generalizing the refresh mechanism to all roles is a separate epic.
- **Push of secrets into the VM.** API keys continue to come from `forward_env` at session start, not from the rendered CCR config.
- **Watching `.clauded.yaml` for live edits.** No file-watcher; refresh runs once per `clauded` invocation. Long-running sessions don't pick up YAML edits made after launch.
- **Custom-renderer plugins.** No user-pluggable hooks into the rendering pipeline.

## Non-functional Considerations

- **Renderer drift is a real risk.** The drift-detector test (AC-008) is non-negotiable. A second renderer that subtly differs from the first reintroduces exactly the kind of "is my config actually applied?" anxiety this epic exists to remove.
- **VM communication channel.** Pushing files into the VM via `limactl copy` or SSH `cat >` are both viable; pick the one with the lower per-call overhead and document in the implementation. Whatever path is chosen, the host CLI must already have a working SSH/limactl channel established (it does â€” the harness launch uses one).
- **CCR restart cost.** A bounce typically takes ~1-2 seconds. Combined with the file push, the 3-second budget (R5) is achievable but tight. If implementations find the budget consistently violated, prefer "skip restart when config unchanged" (R7) over relaxing the budget.
- **Error path stays loud.** The "harness must not launch with stale config when new config is invalid" rule (R6) is the contract. A silent fallback to old config would make `.clauded.yaml` edits non-deterministic and undo the value of this feature.

## Open Questions

1. **Where does the renderer actually live?** Option A1: a `clauded.ccr.render` module called both directly by the CLI and by an Ansible `action_plugin`. Option A2: drop role-side rendering entirely; Ansible installs the package, the CLI always renders. A1 keeps current behavior viable on first provision; A2 simplifies the dataflow but means a fresh VM has no CCR config until the first `clauded` launches the harness â€” usable but more surprising.
2. **CCR restart trigger.** Bumping a `~/.claude-code-router/config.fingerprint` file the wrapper's watcher checks, vs. sending CCR a signal, vs. relying on existing watcher logic with a config-mtime check. The watcher currently triggers on token changes; extending it to config changes is a small wrapper edit. Decide during implementation.
3. **Push mechanism**: `limactl copy` (per-file invocation, slower) vs. heredoc-over-SSH (one connection, faster but uglier). The CLI already has SSH plumbing for the harness launch â€” likely cheaper to reuse.
4. **First-launch UX**: if A2 is chosen and CCR's config is rendered only on first `clauded`, the wrapper's health probe must tolerate a "no config yet" state for the first ~2 seconds. Not hard, but worth thinking through.
